# 実験案 v4：顧客のジョブ ── ライフサイクル軸（時間で変わるジョブ）

> 実験ラベル：**v4 / ライフサイクル軸**
> 作成日：2026-04-25
> 視点：「親と子のジョブ」は時間とともに**質的に変わる**。初日の主要ジョブは「とにかく動く成功体験」。1 週後は「サイクルが回る感触」。1 ヶ月後は「自分たちのゲーム化」。半年後は「友達コミュニティ」。1 年後は「次の世代に渡す」。**ジョブを時間で切る**ことで、プロダクトのフェーズ別優先順位が見える。
> 動機：v1（外）／v2（内）／v3（事故）に対して、v4 は「**経過時間**」軸。プロダクトを「フェーズの連続」として設計する。

---

## 0. なぜライフサイクル軸か

「親が子と一緒にゲームを作る」というジョブは、家族にとって**長期的なプロジェクト**。半年〜1 年継続して初めて成果が出る。だが、現状のジャーニーは時間軸を持っていない——どのジャーニーも「いつ起きるか」を区別していない。

実際の家庭では：
- **Day 1**：とにかく「動いた！」体験が必要。複雑な機能は要らない。
- **Week 1**：「サイクルが回る」を体感する。1 日に何周も回せる感触。
- **Month 1**：「自分たちのゲーム」化が始まる。オリジナル要素が積み上がる。
- **Month 3-6**：友達フィードバックループが起動する。家族の外との接続。
- **Year 1**：作品が完成する。次の世代（弟妹、別のプロジェクト）への引き継ぎ。

それぞれのフェーズで**主役のジョブ**と**最大のリスク**が違う。

---

## 1. 5 つのフェーズ

| フェーズ | 期間 | 主役のジョブ | 最大のリスク |
|---|---|---|---|
| **Phase 0: First Light** | Day 0-1 | 動いて／変えて／動く体験 | 起動できずに終わる |
| **Phase 1: Loop Ignition** | Day 1 - Week 1 | 数分のサイクルが何周も回る | 1 周目が長すぎて 2 周目が来ない |
| **Phase 2: Ownership** | Week 2 - Month 1 | オリジナル要素の積み上げ | 改造が他所を壊して止まる |
| **Phase 3: Outreach** | Month 1 - Month 6 | 友達フィードバックループ起動 | 配信が壊れる／友達が遊ばない |
| **Phase 4: Inheritance** | Month 6 - Year 1+ | 作品完成、次への継承 | エンディングまで遊び切れない／興味が蒸発 |

---

## 2. フェーズ別ジョブ

### Phase 0: First Light（最初の 1 時間）

> **L0.1**：インストールせずに、5 分以内にゲームが起動する  
> **L0.2**：起動した瞬間に、子が「なにか変えてみたい」と思える形を見せる  
> **L0.3**：1 個でも変更が画面に反映される瞬間を作る

**設計に降ろすと**：
- L0.1 → スマホブラウザで URL を開けば即起動（ビルド済み版が selector に並ぶ）
- L0.2 → 起動直後の画面が「自分のキャラが動く」具体的なゲーム
- L0.3 → リソースエディター（Tilemap）で 1 タイル置いて Run までを 1 分で

**最大の差別化**：「セットアップ」が無いこと。Roblox Studio はインストールが要る／Scratch はサイト登録／RPG Maker は購入。CQP は URL 一発。

---

### Phase 1: Loop Ignition（1 日〜1 週間）

> **L1.1**：1 セッション（30 分〜1 時間）の中で 5 周以上ループが回る  
> **L1.2**：ループの 1 周が「子の集中の単位」（数分）を超えない  
> **L1.3**：壊れて止まるリスクがほぼゼロ（ガードレール初動）

**設計に降ろすと**：
- L1.1 → AI 翻訳 + ヘッドレス検証 + 開発版反映を**合計 5 分以内**に保つ
- L1.2 → 「敵強すぎ」→「直ったよ」→「もう一回戦って」の往復を 3 分以内
- L1.3 → 壊れた版を子に届けない

**最大の差別化**：1 周のテンポ。Roblox Studio は 1 周がそもそも長い／RPG Maker も同じ。CQP の「数分で 1 周」が**この時期の継続率を決める**。

---

### Phase 2: Ownership（2 週目〜1 ヶ月）

> **L2.1**：オリジナルの要素（マップ、敵、セリフ、技、見た目、音）が積み上がる  
> **L2.2**：「ベースの Block Quest 感」が薄れていく実感  
> **L2.3**：見た目・音まで自分たちで触れる

**設計に降ろすと**：
- L2.1 → 一覧表のジャーニー（CJ02-CJ07, CJ23-CJ24, CJ09-CJ14）が網羅的に踏める
- L2.2 → 改造前後の比較（履歴 UI）でオリジナル比率が見える
- L2.3 → Code Maker で `code-maker.zip` を持ち出して見た目・音をさわれる（CJ26）

**最大の差別化**：見た目までの所有感。Roblox はアセット使い回し／SMM2 はマリオ感が抜けない。CQP はピクセルアートからオリジナルにできる。

---

### Phase 3: Outreach（1 ヶ月〜半年）

> **L3.1**：友達に URL を送って遊んでもらう  
> **L3.2**：友達のフィードバックを家族のループに取り込む  
> **L3.3**：「届いた／届かなかった」を事実で見直す

**設計に降ろすと**：
- L3.1 → スマホ URL 即プレイ（CJ21）
- L3.2 → CJ22（友達のフィードバックを反映）の往復が機能
- L3.3 → 実公開アクセスログ（CJ43）

**最大の差別化**：家庭外との接続のしやすさ。Scratch はコメント文化／Roblox は社会圧の強さ／RPG Maker は配信ハードル高。CQP は URL × 数分の修正サイクルで「友達と作る」フェーズを開ける。

---

### Phase 4: Inheritance（半年〜1 年以上）

> **L4.1**：作品（自分たちのゲーム）を最後まで遊び切る  
> **L4.2**：エンディングを自分たちの言葉で書く  
> **L4.3**：弟妹・別プロジェクトに「同じ仕組みを使える」と感じる  
> **L4.4**：休んでも戻ってこられる

**設計に降ろすと**：
- L4.1 → CJ42（最後までやり切る）
- L4.2 → CJ30（エンディングを書く）
- L4.3 → 教材版（`STUDENT AREA`）のテンプレが他プロジェクトでも再利用可能
- L4.4 → セーブデータ・本番版を archive（v3 の F5 と整合）

**最大の差別化**：作品を「終わらせられる」こと。Roblox は終わりがない／Scratch は完結性が弱い／SMM2 はステージ単位で完結しない。CQP は RPG として最後まで遊べる骨格を仕組みで保証。

## 3. フェーズ × ジョブのマトリクス

|  | Phase 0 | Phase 1 | Phase 2 | Phase 3 | Phase 4 |
|---|---|---|---|---|---|
| Job1（コミュニケーション） | — | 子の言葉を翻訳できる | 親子で見た目・音を相談 | 友達の声を子に渡す判断 | 「これでいい？」を聞き合う |
| Job2（好循環） | 1 周目を回す | 数分の 1 周を何周も | 改造の積み上げ | 友達ループ起動 | サイクルの集大成 |
| Job3（作品感） | — | — | オリジナル比率上昇 | 友達に「すごい」と言われる | 作品としての完成 |
| Job4（楽しむ） | 動いて喜ぶ | 何周も楽しむ | 「自分のキャラ」感 | 友達と笑う | エンディングを見る |
| Job5（学ぶ） | 副産物 | 副産物 | 副産物 | 興味があれば深掘り | 弟妹に教える |

各フェーズに「主要ジョブ」が 1〜2 個ある。**他のジョブはそのフェーズでは無理に満たそうとしない**のがポイント。

## 4. フェーズ別「最重要ジャーニー」マッピング

| フェーズ | 主要ジャーニー | 副次ジャーニー |
|---|---|---|
| Phase 0 | 起動成功（ジャーニー化されてない、新規 CJ46？） | CJ01（タイル配置） |
| Phase 1 | CJ01-CJ07（マップ）, CJ08（敵調整）, CJ12（バグ報告） | CJ23（スプライト）, CJ24（音） |
| Phase 2 | CJ09（セリフ）, CJ10（敵追加）, CJ13（呪文）, CJ27（ストーリー分岐） | CJ15-CJ20（演出）, CJ31-CJ34（承認） |
| Phase 3 | CJ21（友達に見せる）, CJ22（フィードバック反映）, CJ43（アクセスログ） | CJ28（新エリア）, CJ29（バランス） |

**ガードレールは全フェーズで等しく重要**。それ以外のジャーニーには「いま生きてる時期」がある。

## 5. フェーズ別の落とし穴

| フェーズ | 落とし穴 | 防ぎ方 |
|---|---|---|
| Phase 0 | セットアップで離脱 | スマホ URL 一発、起動成功率 100% |
| Phase 1 | 1 周目が長くて 2 周目が来ない | AI 翻訳 + 検証 + 反映を 5 分以内に |
| Phase 2 | 改造が他所を壊して止まる | ガードレール CJ35-CJ41 |
| Phase 3 | 友達が遊んでくれず子が落ち込む | アクセスログで親(観察者) が事実判断 |
| Phase 4 | エンディングが書けず作品が終わらない | CJ30, CJ42 の仕組みでやり切れる構造を担保 |

## 6. このバージョンを採用するときに変わること

- ジャーニーが**フェーズ別グループ**で整理される（journeys-v4 参照）
- マーケティング素材が**フェーズごとの訴求**に分かれる
  - Phase 0 訴求：「インストール不要、5 分で起動」
  - Phase 1 訴求：「数分で 1 周、子の集中が切れない」
  - Phase 2 訴求：「ピクセルから音まで自分の手で」
  - Phase 3 訴求：「友達と一緒に作るループ」
  - Phase 4 訴求：「作品を最後まで遊び切れる」
- プロダクトのリリース・改善優先順位がフェーズ別になる：「Phase X の体験を完成させる」と区切る
- ロイヤリティ評価軸が「**何ヶ月続いたか**」「**どのフェーズに到達したか**」になる

---

## 7. ライフサイクル KPI 候補

| フェーズ | KPI |
|---|---|
| Phase 0 | 起動成功率（URL を開いた家庭のうち、5 分以内にゲームが動いた割合） |
| Phase 1 | 1 セッションあたりの周回数（中央値） |
| Phase 2 | 1 ヶ月後のオリジナル比率（既存 Block Quest の何割が書き換わったか） |
| Phase 3 | 友達 URL の到達率／プレイ率／フィードバック到達率 |
| Phase 4 | エンディング到達率／継続月数の中央値 |

これらが**親(観察者) ダッシュボード**に乗ると、家族自身が自分たちの位置を把握できる。
