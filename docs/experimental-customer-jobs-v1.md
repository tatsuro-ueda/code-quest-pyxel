# 実験案 v1：顧客のジョブ ── 競合プロダクト軸で書き直す

> 実験ラベル：**v1 / 競合プロダクト軸**
> 作成日：2026-04-25
> 視点：「親子で自分たちのゲームを作って遊ぶ」というジョブを、Scratch・Roblox Studio・Super Mario Maker・RPG Maker・GDevelop・Bitsy・Pyxel Code Maker 単独・「vibe coding（AI 自然言語生成）」がそれぞれ**どこまで**・**どう**果たすかを並べ、Code Quest Pyxel（以下 CQP）が**唯一カバーする領域**を浮き彫りにする。

---

## 0. ベースライン：このジョブを巡る「ある家庭の現実」

> 子ども（小学生）がゲームを作りたいと言う。  
> 親はプログラミングはできる。  
> 子どもは「このスライム強すぎ」「もっとカッコいい技ほしい」と要望を出す。  
> 親は「子どもの主体性を奪わずに、でも一緒に作品を完成させたい」と思っている。  
> 友達に見せられる形にしたい。  
> でも家庭時間は限られている。

このベースラインが、以下の競合と CQP を同じ土俵で比べるための「同一ジョブ」になる。

---

## 1. 競合プロダクト・マトリクス

### 1.1 機能と提供範囲

| プロダクト | 何を作れるか | 子どもが触る粒度 | 親の関与 | 配信 | コミュニティ |
|---|---|---|---|---|---|
| **Scratch** | 2D ゲーム・アニメ・ストーリー | ブロックプログラミング | 子ども単独可 / 親は補助 | scratch.mit.edu に共有 | リミックス文化 |
| **Roblox Studio** | 3D ゲーム | Lua 直書き | 親はほぼ関わらない | Roblox プラットフォーム | 巨大、課金経済（Robux） |
| **Super Mario Maker 2** | マリオ風ステージのみ | パーツの配置 | 子ども単独可 | Nintendo Online | コース ID 共有 |
| **RPG Maker MV/MZ** | 2D RPG | UI 操作 + JS スクリプト | 親も関与しないと深くは作れない | 自分で配布／Steam | itch.io / Steam |
| **GDevelop / Construct** | 2D ゲーム全般 | ノーコード（イベントシート） | 親が共同作業しやすい | HTML5 / モバイル書き出し | itch.io |
| **Bitsy** | ミニ世界・対話 | パーツ配置 | 親はほぼ不要 | Web 埋め込み | itch.io、ブログ |
| **Pyxel Code Maker 単独** | レトロ素材・音 | ドット絵／チップチューン編集 | 親がコードと統合 | 自前ビルド | 個別共有 |
| **vibe coding（生成 AI）** | 何でも | 自然言語で要望 | 親が AI と話す | 親が配信を組む | プロダクト依存 |
| **Code Quest Pyxel** | 既存 RPG を**育てる** | リソースエディタ＋承認キュー | 親＝AI 翻訳係、子＝ハンドル | スマホ URL 即配信 | 友達フィードバック |

### 1.2 「ベースライン家庭」での実体験

| プロダクト | 親子の関係 | 修正サイクル | 「自分たちのゲーム」感 | 友達への共有 |
|---|---|---|---|---|
| Scratch | 子が主、親が困ったら助ける | 子の手で数十秒〜数分 | 「Scratch で作った」感 | URL は出るが PC ブラウザ前提 |
| Roblox Studio | 子主体だが**約 80% は完成しない**（学習曲線で離脱） | 子だと数日〜未完 | 完成すれば強い／途中放棄が多い | プラットフォーム内、課金圧 |
| Super Mario Maker 2 | 完全に子主体 | 数分 | 「マリオのステージ」感 | コース ID で世界中 |
| RPG Maker | 親が主、子は見学になりがち | 機能追加 1 つに半日 | RPG として完成度高いが時間がかかる | 自前配信 |
| GDevelop | 親子で半々 | 数十分 | 中庸 | itch.io |
| Bitsy | 子主体 | 数分 | ミニ作品感 | Web 埋め込み |
| Pyxel Code Maker 単独 | 親が大半 | 素材は数分、コード統合は数十分 | 完成しないと出ない | 自前 |
| vibe coding | 親が大半 | 数分（ただし壊れる頻度高） | 「親が AI と作った」感 | 親が組まないと出ない |
| **CQP** | **エンジンとハンドルが分離** | 数分（子の集中が切れない） | 「ぼくたちのゲーム」 | スマホ URL で即 |

## 2. ジョブを「競合では届かない領域」として再定義する

各ジョブを、競合がどこまで果たし、どこから先が CQP の領域かで定義する。

### Job1：子どもとコミュニケーションする（先生／生徒化を回避）

> 子どもと「一緒に作る仲間」になりたい。先生／生徒にはなりたくない。

| 競合の対応 | 限界 |
|---|---|
| Scratch：子どもが自分で作るので親は教える側になりがち | 親が知識マウント → 先生化 |
| Roblox：親が関わると Lua の壁。関わらないと完成しない | どちらに転んでも対等関係を維持しにくい |
| RPG Maker：機能が多くて親が「正しい使い方」を教えがち | 先生化しやすい |
| vibe coding：親が AI と話すので、子は「お客さん」になりがち | 子の意思決定が消える |

**CQP がカバーする領域**：
- 親は「AI に翻訳する」だけで「何を直すか」は決めない
- 「開発版／本番」の遊び比べで、子どもが**体感**を根拠に選ぶ
- 親の理屈で押し切れない仕組み（承認キューの却下フロー）

→ **Job1 を競合と差別化する固有要素**：「親が直す権限を持たないシステム」「子どもの体感が決定根拠になる遊び比べ機構」

---

### Job2：好循環な開発を実現する（数分で 1 周回る）

> 「作る → 遊ぶ → 直す → また遊ぶ」を親子で回したい。

| 競合の対応 | 限界 |
|---|---|
| Scratch：子の手で数十秒で回る（だが親は外野） | 親子協働の好循環ではない |
| Roblox：1 周がそもそも回らない（80% 未完成） | ループ自体が成立しない |
| SMM2：ステージ修正は数分（だが RPG ではない） | 内容の変更に限界 |
| RPG Maker：1 周が半日 | 子の集中が切れる |
| vibe coding 単独：数分で回るが**壊れた版が子に届く** | 逆フィードバックで好循環が破壊される |

**CQP がカバーする領域**：
- AI 修正 → ヘッドレス検証 → 開発版カード → 子が遊び比べ → 承認 → 本番昇格
- 「壊れた版を子どもの画面に届けない」ガードレール（CJ35-CJ41）
- 既存 Block Quest を**ベース**にして「ゼロから」ではなく「育てる」

→ **Job2 を競合と差別化する固有要素**：「壊れない好循環」「親の AI 翻訳と子のハンドルが分離した状態でループが閉じる」

---

### Job3：アマチュアレベルのアウトプットを出す

> 友達に見せて「すごい！」と言われたい。

| 競合の対応 | 限界 |
|---|---|
| Scratch：見た目はスクラッチ感が抜けない | 「ちゃんとしたゲーム」には見えにくい |
| Roblox：見た目は強い（完成すれば） | 80% 未完成、課金圧、Roblox 内に閉じる |
| SMM2：見た目は完璧（マリオ）／オリジナル感ゼロ | 「自分たちのゲーム」と言いにくい |
| RPG Maker：見た目は強い／時間と労力が要る | 完成までもたない |
| Bitsy：味わいはあるが「ミニ作品」 | 友達に「すごい」と言わせるボリュームではない |

**CQP がカバーする領域**：
- レトロピクセル（Pyxel）の表現力で「作品感」が出る
- 親子が Code Maker で**見た目・音**まで自分で触れる
- AI が build / verify を担当 → 親子が美術と物語に集中できる

→ **Job3 を競合と差別化する固有要素**：「短時間で作品感が出る土台」「美術と物語の意思決定を親子に残す分業」

---

### Job4：ゲームを楽しむ（子）

> 自分たちが作ったゲームを最後まで遊び切れる。

| 競合の対応 | 限界 |
|---|---|
| Scratch：作ったものを自分で遊ぶ | RPG として最後まで遊べる構成は稀 |
| Roblox：他人の作品を遊ぶのが中心になる | 自分が作ったものを遊ぶ体験は弱い |
| SMM2：自分のステージで遊べる | RPG の「冒険を最後までやり切る」体験ではない |
| RPG Maker：最後まで遊べる作品は作れる | 半年〜1 年仕事 |

**CQP がカバうする領域**：
- 既存 RPG を改造する形式 → 最初から最後まで遊べる土台がある
- 改造を重ねるほど「自分たちのゲーム」化（CJ30, CJ42）
- エンディングまで含めた「やり切れる構造」を仕組みで保証（CJ42）

→ **Job4 を競合と差別化する固有要素**：「改造を重ねても遊び切れる骨格を保つ仕組み（ガードレール 7 種）」

---

### Job5：（興味があれば）プログラミングを学ぶ（子）

> 強制ではなく、興味の延長で技術に触れたい。

| 競合の対応 | 限界 |
|---|---|
| Scratch：これが学習目的のプロダクト | 学習が目的化、ゲーム作品化が二の次 |
| Roblox：Lua を書ける子は学べる | 学習曲線で挫折率高い |
| RPG Maker：JS スクリプトで深掘り可 | 深掘りしないとオリジナリティ出ない |

**CQP がカバーする領域**：
- 子は**触りたいときだけ**コードに触れる（Code Maker の audio/sprite が入り口）
- `STUDENT AREA` の明示でコア領域とユーザー領域が分かれている
- 学習自体は副産物。主役はあくまで「自分たちのゲームを作る」

→ **Job5 を競合と差別化する固有要素**：「学習を目的化しない／興味のときだけ深掘りできる段差設計」

---

### 【新規】Job6：友達フィードバックを家族の好循環に取り込む

> 友達が遊んだ反応を、親子の次のサイクルに即座に反映したい。

| 競合の対応 | 限界 |
|---|---|
| Scratch：コメントは付くが PC 前提、即プレイは弱い | 友達がスマホで触れない |
| Roblox：コミュニティは強いが家族の枠を越える社会圧 | 親が見守りにくい |
| SMM2：コース ID 共有はできる | フィードバックを反映する機構がない |
| RPG Maker：自前配信、即プレイは難しい | 友達が試遊するハードルが高い |

**CQP がカバーする領域**：
- スマホ URL 即プレイ（インストール不要・登録不要）
- 友達のフィードバック →「直して！」を判断するのは子（CJ22）
- 実公開アクセスログで「届いた／届かなかった」を事実で見直せる（CJ43）

→ **Job6 を競合と差別化する固有要素**：「家庭の外（友達）からの即時フィードバックを家族の好循環に閉じ込める導線」

---

### 【新規】Job7：親の暴走をシステムで止める

> AI で何でも数分で直せる時代に、親が「しゃしゃり出す」ことを子のシステムで防ぎたい。

| 競合の対応 | 限界 |
|---|---|
| すべての競合：システム的にこのリスクを扱っていない | 親の自制に依存 |

**CQP がカバーする領域**：
- 承認キュー：親の修正は**必ず開発版**に行く。本番昇格は子の承認後
- 「もどして」も承認キューで回る（CJ34）
- 役割分担表：親は AI 翻訳・技術提供、子はハンドル

→ **Job7 を競合と差別化する固有要素**：「親の善意の暴走をシステムで止める承認機構」

---

## 3. CQP の独自領域を一行で

> **「親が AI で直すのが速いほど壊れる『親子の対等関係』を、システムで保ちながら、家庭の外まで好循環を広げる土台」**

---

## 4. 採用する優先順位（このバージョンの主張）

1. **Job7（親の暴走をシステムで止める）と Job1（先生／生徒化回避）が、競合と最も差別化できる**
2. Job2（好循環）と Job6（友達フィードバック）はスピード × ガードレールで実現
3. Job3（作品感）・Job4（やり切る）は土台ありき → 既存 Block Quest の改造方式
4. Job5（学習）は副産物。一切目的化しない

---

## 5. このバージョンを採用するときに変わること

- ジャーニー本文の Before に「Scratch でやった場合」「Roblox でやった場合」など**具体的な競合シナリオ**が並ぶ（journeys-v1 参照）
- 一覧表に「対応する競合」「競合の限界」列が増える
- product-requirements ／ structure-design に「Job7 を満たす承認キューの不可侵条件」が降りる
- マーケティングの軸が「AI で速い」ではなく「**親が暴走できない**親子の対等関係」になる
