# カスタマージャーニーgherkin: Narrative（ダイアログ・ストーリー・ナビ）

- 対象カスタマージャーニー: CJ09, CJ14, CJ27, CJ30
- 根拠: [`customer-journeys.md`](./customer-journeys.md)
- 中心テーマ: セリフ、会話分岐、目的地ガイド、エンディング
- 読み方: `Feature` は何を約束するか, `Rule` は何を壊してはいけないか, `Scenario` は画面や動作でどう確かめるか
- 実装状況の読み方: `実装済み` は現行コードとテストで確認できる, `部分実装` はコアはあるが文中の約束を全部は満たしていない, `未実装` は目標, `時間目標` は現状の目標値であり自動保証ではない

---

## CJG09: セリフを変えたい

この `CJG09` では、「NPC のセリフを変えたら、その町らしさやキャラらしさがすぐゲームに出る」を約束する。

実装状況:
- `実装済み`: 会話データは `assets/dialogue.yaml -> src/generated/dialogue.py` で管理され、実行時に scene 名から読む
- `時間目標`: `5分以内` は現状の目標値であり、自動計測はしていない

```gherkin
Feature: CJG09 セリフを変えたい
  子どもが世界の言葉も自分たちのものだと感じるには
  セリフ変更がすぐ反映され
  他の会話まで壊してはいけない
```

```gherkin
Rule: 変えたセリフはすぐそのNPCに反映される
  どこを変えたのか分からないと、物語づくりの手応えがなくなる

  Scenario: NPCのセリフを変更するとテストプレイで即座に反映される
    Given 町のNPCに「ようこそ」というセリフが設定されている
    When 親がAIに「この町人のセリフを冒険者っぽくして」と頼む
    And AIがダイアログデータを修正する
    And Run する
    Then そのNPCに話しかけると新しいセリフが表示される
    And 修正から確認まで5分以内で完了する
```

```gherkin
Rule: 1人の変更が他の会話に漏れない
  村人Aを直したら村人Bまで変わる、を作ってはいけない

  Scenario: セリフ変更が他のダイアログを壊さない
    Given 特定のNPCのセリフを変更した
    When Run して他のNPCに話しかける
    Then 変更していないNPCのセリフは元のまま表示される
```

```gherkin
Rule: 子どもが読める形で表示される
  読めない文字や文字化けは、物語への参加を止めてしまう

  Scenario: ひらがな中心でセリフが表示される
    Given セリフデータにひらがな中心のテキストがある
    When NPCに話しかける
    Then セリフが想定した表記で正しく表示される
    And デフォルトフォントで文字化けしない
```

## CJG14: マップが広すぎて迷う

この `CJG14` では、「次にどこへ行けばいいかが、親に聞かなくても分かる」を約束する。

実装状況:
- `未実装`: 現在のゲームには汎用の目的地ガイド UI やナビ表示はない
- `部分実装`: ランドマーク強調表示や会話による導線は一部あるが、次目的地ガイドとは別物

```gherkin
Feature: CJG14 マップが広すぎて迷う
  子どもが広い世界でも一人で進めるには
  次の目的地が分かり
  物語の進行に合わせて案内が変わらなければならない
```

```gherkin
Rule: 今行くべき場所が画面上で分かる
  目的地が分からないと、冒険ではなく迷子になる

  Scenario: 次の目的地へのガイドが表示される
    Given ストーリー進行に応じた次の目的地が設定されている
    When 子どもがフィールドを歩いている
    Then 次の目的地の方向または場所が何らかの方法で示される
```

```gherkin
Rule: 案内は物語の進行に合わせて更新される
  前の目的地を指し続けると、子どもは何を終えたのか分からなくなる

  Scenario: ガイドがストーリー進行に連動する
    Given 目的地Aに到達してイベントを完了した
    When フィールドに戻る
    Then ガイドが次の目的地Bを示すように更新される
```

```gherkin
Rule: 案内は答えを全部言いすぎず、進む方向だけは失わせない
  何も分からないのも、全部説明されすぎるのも、冒険の楽しさを弱める

  Scenario: ガイドが進む方向を示しつつ探索の余地を残す
    Given 子どもがまだ目的地に到達していない
    When ガイドが表示される
    Then 子どもは次に向かう大まかな方向を理解できる
    And 謎解きや探索の答えが最初から全部表示されない
```

## CJG27: ストーリーの分岐を作る

この `CJG27` では、「はい・いいえのような選択が、本当に話や進み方を変える」を約束する。

実装状況:
- `部分実装`: `StructuredDialogRunner` 自体は choices/variants を持つ
- `部分実装`: 現在の本編で強く使っているのは教授導線など限られた分岐で、汎用の分岐編集体験にはなっていない

```gherkin
Feature: CJG27 ストーリーの分岐を作る
  子どもが自分で物語を選んだと感じるには
  選択肢が表示され
  選んだ結果が会話と進行の両方に反映されなければならない
```

```gherkin
Rule: 選べる会話が画面に出る
  選択肢が裏では存在していても、子どもが選べなければ分岐にならない

  Scenario: ダイアログに選択肢を追加できる
    Given NPCとの会話データがある
    When 親がAIに「王様の依頼にはい/いいえの選択肢を追加して」と頼む
    And AIがダイアログデータに選択肢を追加する
    Then 会話中に選択肢が表示される
    And 子どもがその場で選べる
```

```gherkin
Rule: 選んだ言葉で会話の続きが変わる
  どちらを選んでも同じ返事なら、選んだ意味がない

  Scenario: 選択肢によって会話の続きが変わる
    Given 王様の依頼に「はい」「いいえ」の選択肢がある
    When 「はい」を選ぶ
    Then 依頼を受けた旨のセリフが表示される
    When 「いいえ」を選ぶ
    Then 断った反応のセリフが表示される
```

```gherkin
Rule: 選択結果はその後のゲームにも残る
  会話だけ変わって進行が同じでは、分岐の実感が弱い

  Scenario: 選択結果がゲーム状態に反映される
    Given 選択肢による分岐がフラグと紐づいている
    When 子どもが「いいえ」を選ぶ
    Then 対応するフラグが設定される
    And 以降のゲーム進行がそのフラグに応じて変化する
```

## CJG30: エンディングを自分たちで書く

この `CJG30` では、「最後の言葉や名前まで自分たちの作品にできる」を約束する。

実装状況:
- `部分実装`: エンディング本文は dialogue データ経由で差し替えられる
- `未実装`: クレジットに名前を入れる専用機能は現状ない

```gherkin
Feature: CJG30 エンディングを自分たちで書く
  子どもが「これは自分たちのゲームだ」と感じるには
  最後のメッセージやクレジットを変えられて
  その変更が本編を壊してはいけない
```

```gherkin
Rule: エンディングの言葉を変えられる
  最後だけ誰かの言葉のままだと、作品の締めくくりを自分たちのものにできない

  Scenario: エンディングのセリフを変更できる
    Given エンディング画面に表示されるテキストがある
    When 親がAIに「エンディングのセリフを変えて」と頼む
    And AIがエンディングデータを修正する
    And Run してゲームをクリアする
    Then 変更後のセリフがエンディング画面に表示される
```

```gherkin
Rule: 名前を入れて自分たちの作品だと分かる
  クレジットに自分の名前が出ることが、完成の実感につながる

  Scenario: クレジットに名前を入れられる
    Given エンディングにクレジット表示がある
    When クレジットのテキストに子どもの名前を追加する
    And Run してエンディングに到達する
    Then クレジットに子どもの名前が表示される
```

```gherkin
Rule: 終わり方を変えても本編は最後まで遊べる
  エンディングだけ直したつもりで本編が壊れると、完成体験がなくなる

  Scenario: エンディング変更がゲーム本編を壊さない
    Given エンディングのデータを変更した
    When Run してゲーム本編をプレイする
    Then エンディング以外のゲーム進行が正常に動作する
```

## 共通条件

この共通条件では、「会話や物語のデータを直しても、子どもに見える読みやすさと進行のつながりが壊れない」を約束する。

実装状況:
- `実装済み`: dialogue の整合性、scene ID のカバレッジ、ページ送りはテストがある

```gherkin
Feature: 共通条件 Narrative
  物語を安心して編集するには
  会話データがつながっていて
  Code Maker でも本編でも正しく読めなければならない
```

```gherkin
Rule: 会話データは切れずにつながる
  空のセリフや行き先のない選択肢は、物語を途中で壊してしまう

  Scenario: ダイアログデータの整合性
    Given ダイアログデータに定義されたすべての会話
    Then 各会話のテキストが空でない
    And 参照されるNPCや場所のIDが有効である
    And 選択肢がある場合、各選択肢に遷移先が設定されている
```

```gherkin
Rule: 会話は Code Maker でも本編でも同じように読める
  編集画面では読めるのにゲームで壊れる、を作ってはいけない

  Scenario: ダイアログ表示が Code Maker でも正常に動作する
    Given Code Maker 上でゲームを実行する
    When NPCに話しかける
    Then ダイアログが正常に表示される
    And 外部ファイル参照に依存せずに動作する
```
