# 顧客課題：親子でゲーム開発の好循環を回す

- 作成日: 2026-04-10
- 対象プロジェクト: Pyxel版 Block Quest
- 主役: **親**（プログラミングが出来る、子どもを持つ大人）と**子ども**

---

## 1. ジョブ

> **ジョブ1: 子どもとコミュニケーションする**
> 子どもと「一緒に作る仲間」になりたい。先生／生徒にはなりたくない。

> **ジョブ2: 子どもと好循環な開発を実現する**
> 「作る → 遊ぶ → 直す → また遊ぶ」を親子で回したい。

> **ジョブ3: アマチュアレベルのアウトプットを出す**
> 回した結果が「ちゃんとしたゲーム」に見えてほしい。友達に見せて「すごい！」と言われたい。

---

## 2. 好循環とは何か

このプロジェクトの好循環は、親子の間を行き来する内側のループと、友達にまで広がる外側のループで構成される。

**作るループ**: 子どもがTilemapエディタでマップを描き、Runして歩いてみる。「ここに木を置こう」「水で道を塞いでみよう」。操作→結果が数十秒で回る。

**直すループ**: 子どもがテストプレイして「ここ変だよ」「この敵強すぎ」と言う。親がAIに頼んで数分で直す。「直ったよ、もう一回やってみて」。子どもがまた遊んで次のフィードバックを出す。

**広がるループ**: 友達にURLを送る。友達がスマホで開いて即プレイ。「ここ難しい」「ここ面白い」とフィードバックが返ってくる。親がその場でAIに頼んで直す。「直ったよ、もう一回やってみて」。フィードバックの出し手が子ども一人から友達にまで広がることで、好循環の回転がさらに速くなる。

```
  子ども                          親
  ┌──────────┐               ┌──────────┐
  │ マップを描く │──→ Run ──→   │          │
  │ テストプレイ │              │ AIに頼んで │
  │ 「ここ変だよ」│──→ 報告 ──→ │ 数分で直す │
  │ 「直った！」 │←── 反映 ←── │          │
  └──────────┘               └──────────┘
       ↑                          │
       └──── 「もう一回やって」 ←───┘

  友達（スマホ）                    親
  ┌──────────┐               ┌──────────┐
  │ URLを開く   │              │          │
  │ スマホでプレイ│              │ AIに頼んで │
  │ 「ここ難しい」│──→ 報告 ──→ │ 数分で直す │
  │ 「お、変わった」│←── 再共有 ←│          │
  └──────────┘               └──────────┘
       ↑                          │
       └──── 「もう一回やって」 ←───┘
```

子どもは「テストプレイヤー兼共同制作者」であり、親は「子どもの声をAIに翻訳するプロデューサー」。どちらも必要で、どちらも対等。これがジョブ1（先生／生徒にならない関係）を自然に実現する。

さらに、スマホで遊べることでフィードバックの出し手が友達にまで広がる。友達はインストールも会員登録もなしに、URLを開くだけで即プレイできる。この手軽さが「ちょっとやってみて」のハードルを下げ、フィードバックの量と速度を上げる。好循環は回す人が増えるほど速く回る。

---

## 3. なぜ今まで回らなかったか

この好循環は、AIコーディング以前は成立しなかった。

子どもが「この敵強すぎ」と言ったとき、親がコードを探し、理解し、修正し、テストするのに1日以上かかった。翌日に「直したよ」と言っても、子どもの関心はもう別のことに移っている。フィードバックと修正の間が空きすぎて、ループが閉じない。

AIコーディングにより、「ここ変だよ」→「直ったよ」が数分になった。子どもの集中が続いている間にループが1周する。だから何周も回せる。

| | 従来 | AIコーディング後 |
|---|---|---|
| 「この敵強すぎ」→ HP調整 | コードを探して修正（30分） | AIに頼む（3分） |
| 「新しいマップほしい」→ 追加 | 設計→実装→デバッグ（1日以上） | AIに説明→生成→調整（数十分） |
| フィードバック→反映の間隔 | 翌日以降 | **その場で** |

ただし、AIだけでは足りない。子どもが自分の手で変えられる場所がないと、待っているだけの「テスター」になってしまう。Tilemapエディタがあることで、子どもは「遊ぶ人」であり「作る人」でもある状態が保たれる。

---

## 4. ソリューションの構成

4つの要素が揃って初めて好循環が回る。

| 要素 | 役割 | なければどうなるか |
|---|---|---|
| **Tilemap エディタ** | 子どもが自分の手でマップを作る | 子どもが「見ているだけ」になる |
| **AIコーディング** | 親が子どもの声を数分で反映する | フィードバックが翌日になり、ループが閉じない |
| **Pyxel の表現力** | 回した結果がレトロゲームとして成立する | 友達に見せても「ふーん」で終わる |
| **スマホでプレイ可能** | 友達がURLを開くだけで即プレイ→フィードバックが返る | 共有のハードルが高く、フィードバックの出し手が親子だけに閉じる |

好循環が速く回るほどプロダクトの品質は上がり、品質が上がるほど子どもの「もっと作りたい」が強くなり、サイクルがさらに回る。親子のコミュニケーションは、この好循環の副産物として自然に生まれる。

特にスマホ対応は、好循環の「外側の輪」を回すエンジンになる。親子だけで回していたループが、友達→クラスメート→SNSへと広がる可能性を持つ。フィードバックの多様性が上がると「自分では気づかなかった改善点」が見つかり、プロダクトの進化が加速する。

---

## 5. 親の役割の境界線

好循環が速く回せるようになると、親が「しゃしゃり出す」リスクが生まれる。AIコーディングで何でも数分で直せるからこそ、親が勝手に先回りして改善を始めてしまう。これは好循環を壊す最大の落とし穴。

**親の役割は「AIに翻訳する」ことであって、「何を直すか決める」ことではない。**

| やっていいこと | やってはいけないこと |
|---|---|
| 子どもが「ここ変」と言ったものをAIに伝える | 子どもが気にしていないバグを勝手に直す |
| 友達のフィードバックを子どもに伝える | フィードバックを親がトリアージして子どもに知らせない |
| 「どっちを先に直す？」と子どもに聞く | 優先順位を親が決めてしまう |
| 技術的な選択肢を提示する（「AとBどっちがいい？」） | 技術的に「正しい」方を親が選んでしまう |
| 子どもが詰まったときにヒントを出す | 子どもが試行錯誤する前に答えを教える |

好循環のハンドルを握っているのは子ども。親はエンジン（AI翻訳）とガソリン（技術力）を提供するが、行き先を決めるのは子ども。親が行き先まで決めると、子どもは助手席に座るだけの乗客になり、「自分のゲーム」という感覚が消える。

これはジョブ1（先生／生徒にならない）の具体的な実践ルールでもある。「教えない」だけでは不十分で、「決めない」ことが必要。
