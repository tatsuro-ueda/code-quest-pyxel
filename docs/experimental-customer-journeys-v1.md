# 実験案 v1：カスタマージャーニー ── 競合プロダクト軸で書き直す

> 実験ラベル：**v1 / 競合プロダクト軸**
> 作成日：2026-04-25
> 視点：Before に**具体的な競合プロダクトでの体験**を複数並べ、After で CQP 独自領域を立ち上げる。`subgraph Before` を 2〜3 個並列に置き、「どの代替でも届かない価値」が After に残ることを示す。
> 根拠：[`experimental-customer-jobs-v1.md`](./experimental-customer-jobs-v1.md)

---

## 凡例（v1 固有）

- `Before-A` `Before-B` `Before-C` ：複数の代替（既存）行動を並べる
- `After`：CQP の体験
- ノード形式：`[（主体）絵文字 文（タッチポイント）]`
- 主体は `（子ども）` `（親）` `（友達）` `（AI）` `（ビルド）` 等
- 価値は Before 群と After の落差から立ち上がる（解釈軸）

---

## 代表ジャーニー 8 本（厳選）

全 42 本ではなく、**競合との対比が最も鋭くなる 8 本**を厳選して書く。残りは v1 の方針が固まり次第、同方式で展開する。

### CJ01-v1: はじめてのタイル配置

子どもが「マップを変えたい」と言ったとき、各代替プロダクトでは何が起きるか。

**感情**：❌どの代替でも何かが欠ける →❤️親が AI 翻訳役に回り、子どもがハンドルを握ったまま数十秒で世界が変わる

```mermaid
flowchart LR
    subgraph BeforeA["Before-A: Scratch で作る"]
        BA1[（子ども）💡「マップ変えたい」（Scratch エディタ）]
        BA1 --> BA2[（子ども）👆背景画像を差し替える（Scratch エディタ）]
        BA2 --> BA3[（子ども）❌見た目が Scratch 感のまま（Scratch エディタ）]
    end
    subgraph BeforeB["Before-B: Roblox Studio で作る"]
        BB1[（子ども）💡「マップ変えたい」（Roblox Studio）]
        BB1 --> BB2[（子ども）💦Lua とパーツの両方が必要（Roblox Studio）]
        BB2 --> BB3[（子ども）❌80% 未完成の壁にぶつかる（Roblox Studio）]
    end
    subgraph BeforeC["Before-C: 親が main.py を直す"]
        BC1[（子ども）💡「マップ変えたい」（ゲーム中）]
        BC1 --> BC2[（親）💦ソースコードを書き換える（コードエディタ）]
        BC2 --> BC3[（子ども）❌親に説明されて先生／生徒化（対面）]
    end
    subgraph After["After: CQP のリソースエディター"]
        A1[（子ども）💡「マップ変えたい」（ゲーム中）]
        A1 --> A2[（子ども）👆タイルパレットから草を選んで配置（リソースエディター）]
        A2 --> A3[（子ども）❤️Run → キャラがその場所を歩く（ゲーム中）]
    end
```

> **このジャーニーが示す価値**：他の代替はいずれも「子どもの所有感」「即時反映」「親子の対等」のどれかを欠く。CQP はその 3 つを同時に成立させる。

### CJ08-v1: 敵が強すぎる

子どもがテストプレイで「この敵強すぎ！」と言った瞬間、各代替で何が起きるか。

**感情**：❌どの代替でも子の集中が切れる →❤️3 分で戻ってきて一緒に調整できる

```mermaid
flowchart LR

        BA1[（子ども）💡「敵強すぎ！」（Scratch プロジェクト）]

        BA2 --> BA3[（子ども）❌バランス調整は子が独力で抱える（Scratch エディタ）]
    end
    subgraph BeforeB["Before-B: RPG Maker で作っていた場合"]
        BB1[（子ども）💡「敵強すぎ！」（RPG Maker）]
        BB1 --> BB2[（親）💦データベースから敵を探して値を変える（RPG Maker）]
        BB2 --> BB3[（子ども）❌バランス確認のため何度も起動（30 分）（RPG Maker）]
    end
    subgraph BeforeC["Before-C: 親が main.py を直す"]
        BC1[（子ども）💡「敵強すぎ！」（ゲーム中）]
        BC1 --> BC2[（親）💦ENEMIES データを探して HP を変更（コードエディタ）]
        BC2 --> BC3[（子ども）❌30 分後「直したよ」→ もう別のことしてる（対面）]
    end
    subgraph After["After: CQP の AI 翻訳＋承認キュー"]
        A1[（子ども）💡「敵強すぎ！」（ゲーム中）]
        A1 --> A2[（親）👆AI に「スライムの HP 50→30 にして」（AI入力画面）]

        A3 --> A4[（子ども）❤️遊び比べて「こっち！」体感差で承認（選択ページ）]
```

> **このジャーニーが示す価値**：「速い」だけなら vibe coding でできる。CQP の独自領域は「速さ × ガードレール × 子の承認権」が同時に成立すること。

### CJ31-v1: 子どもが変更を承認する

「親が勝手に直さない仕組み」は、競合プロダクトのどこにも無い。

**感情**：❌どの代替でも子は「親が決めたもの」を渡される →❤️開発版と本番を遊び比べて、自分の体感で「こっち！」と決める

```mermaid

        BA1[（子ども）👆自分でブロックを書き換える（Scratch エディタ）]

        BA2 --> BA3[（親子）❌親の知見が活きない／子が独力で詰まる（対面）]
    end
    subgraph BeforeB["Before-B: vibe coding で親が AI と直接やり取り"]
        BB1[（親）👆AI に「スライム弱くして」（AI入力画面）]
        BB1 --> BB2[（親）👆コードに反映（コードエディタ）]
        BB2 --> BB3[（子ども）❌結果だけ渡される → 自分で決めた感覚なし（対面）]
    end
    subgraph BeforeC["Before-C: RPG Maker で親が直接編集"]
        BC1[（親）👆データベースで HP を 50→30（RPG Maker）]
        BC1 --> BC2[（子ども）❓「30 ってどのくらい？よくわかんない」（対面）]

        A1[（親）👆AI に依頼 → 開発版に反映（承認キュー）]
```

> **このジャーニーが示す価値**：「子の意思決定が消えない」を**システムで担保**しているのは CQP だけ。

### CJ35-v1: AI で修正したらエラーが出て動かない

ガードレールの存在は、競合プロダクトと CQP を最も鋭く分ける。

**感情**：❌どの代替でも AI 修正の壊れた版が子の画面に届きうる →❤️ヘッドレス検証で壊れた版は届かない

```mermaid
    end
    subgraph BeforeB["Before-B: RPG Maker で機能追加"]
        BB1[（親）👆プラグインや JS で技を実装（RPG Maker）]
        BB1 --> BB2[（親）💦既存スクリプトと衝突（RPG Maker）]
        BB2 --> BB3[（子ども）❌動作不安定／親が長時間調査（対面）]

    subgraph BeforeC["Before-C: Roblox で Lua 修正"]

        A1[（親）👆AI に依頼 → main.py を修正（AI入力画面）]

        A4 --> A5[（子ども）❤️常に動くゲームだけが届く（ゲーム中）]
```

## このバージョンを採用するときに変わること

- 全 42 ジャーニーが Before に **2〜3 個の競合シナリオを並列**で持つ
- 一覧表に「最も対比が鋭い競合」列が増える
- マーケティング素材は「Scratch ではこうなる、CQP ではこうなる」の比較動画／表
- 親への訴求軸が「親が暴走しない・子の所有感が消えない・ガードレールがある」の 3 点に集約

## 残り 34 本の方針

CJ01-v1 / CJ08-v1 / CJ21-v1 / CJ31-v1 / CJ22-v1 / CJ26-v1 / CJ35-v1 / CJ43-v1 で**全クラスター（開発／デバッグ／共有／承認／ガードレール）の代表**は押さえた。残り 34 本も**この 8 本のテンプレに沿って**展開可能。テンプレ：

1. Before-A：Scratch / Roblox / SMM2 / RPG Maker / GDevelop のいずれか
2. Before-B：別カテゴリの代替（vibe coding 単独 / 自前配信 / Bitsy 等）
3. Before-C：「親が main.py を直す」既存パターン（現家庭の代替）
4. After：CQP の体験
5. 末尾に 1 行「このジャーニーが示す価値」
