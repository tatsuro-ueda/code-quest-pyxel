# カスタマージャーニーgherkin: Map（マップ・タイル操作）

- 対象カスタマージャーニー: CJ01-CJ07
- 根拠: [`customer-journeys.md`](./customer-journeys.md)
- 中心テーマ: Tilemap エディタ、道、森、水辺、迷路、ランドマーク
- 読み方: `Feature` は何を約束するか, `Rule` は何を壊してはいけないか, `Scenario` は画面や動作でどう確かめるか
- 実装状況の読み方: `実装済み` は現行コードとテストで確認できる, `部分実装` はコアはあるが文中の約束を全部は満たしていない, `未実装` は目標, `時間目標` は現状の目標値であり自動保証ではない

---

## CJG01: はじめてのタイル配置

この `CJG01` では、「子どもがタイルを1個置いたら、その変化がすぐゲームに出る」を約束する。

実装状況:
- `部分実装`: tile bank と tilemap の仕組みはある
- `時間目標`: `1分以内` は UX 目標であり、自動保証ではない

```gherkin
Feature: CJG01 はじめてのタイル配置
  子どもが「自分で変えられた」と感じるには
  タイルを置いた結果が
  すぐ見えてそのまま遊べなければならない
```

```gherkin
Rule: 置いたタイルはその場で見える
  クリックしても何も起きないと、子どもは編集できているか分からない

  Scenario: タイルを1個置くと見た目が変わる
    Given Pyxel Code Maker の Tilemap タブが開いている
    And タイルパレットに草・木・水などのタイルが表示されている
    When 子どもがパレットから草タイルを選択する
    And マップ上の任意のマスをクリックする
    Then そのマスが草タイルに変わる
```

```gherkin
Rule: 置いた結果はすぐテストプレイで確かめられる
  編集してから確認まで長いと、親子の試行錯誤が止まる

  Scenario: Run すると置いたタイルの変化がゲームに反映される
    Given Tilemap で1マスのタイルを変更した
    When Run ボタンを押す
    Then ゲームが起動し、変更したマスの見た目がゲーム内でも変わっている
    And 操作開始から確認まで1分以内で完了する
```

```gherkin
Rule: パレットに出るタイルはすべて使える
  見えているのに使えないタイルがあると、子どもには壊れて見える

  Scenario: タイルパレットのタイルがすべてタイルバンクに登録されている
    Given タイルバンクに登録されたタイルIDの一覧がある
    Then パレットに表示されるすべてのタイルがタイルバンクに登録されている
    And 各タイルの16x16ピクセルデータが存在する
```

## CJG02: 道を作る

この `CJG02` では、「子どもが道を置いたら、その道が見た目も通り方もそのままゲーム世界になる」を約束する。

実装状況:
- `部分実装`: 通行判定とオートタイル変種は以前からある
- `未実装`: Code Maker / Resource Editor で見える道と実ゲームが常に一致する保証はまだない
- `時間目標`: `2分以内` は現状の目標値であり、自動計測はしていない

```gherkin
Feature: CJG02 道を作る
  子どもが「ここがみち」と理解して編集するには
  Tilemap エディタで見える道と
  実際に遊ぶ道が同じでなければならない
```

```gherkin
Rule: エディタで見える道がそのままゲームの道になる
  実行時だけ別の形に見えると、子どもは何を編集しているか分からなくなる

  Scenario: 道の見た目が編集画面とゲーム画面で一致する
    Given 子どもが Tilemap エディタで道タイルを連続して配置した
    When Run する
    Then ゲーム内の道のつながり方は Tilemap エディタで見えた形と一致する
    And 実行時だけ別の道の形に補正されない
```

```gherkin
Rule: 道として置いた場所は続けて歩ける
  見た目は道なのに歩けない、逆に道でないのに歩ける、を作ってはいけない

  Scenario: 連続して置いた道をキャラが歩ける
    Given マップ上に通行不可のタイル（山・水など）がある
    When 子どもが道タイルを5マス連続で配置する
    And Run する
    Then キャラが配置した道タイルの上を連続して歩ける
    And 通行不可タイルの上は歩けない
```

```gherkin
Rule: 道づくりは短いサイクルで回る
  描いてから歩いて確かめるまで長いと、子どもが調整しにくい

  Scenario: 道の配置から確認までが2分以内で終わる
    Given Tilemap タブが開いている
    When 子どもが道を描く
    And Run して歩いて確認する
    Then 描画から確認まで2分以内で完了する
```

## CJG03: 森を作る

この `CJG03` では、「木を並べると森になり、通れる場所と通れない場所が自分で分かる」を約束する。

実装状況:
- `部分実装`: 通行不可タイルと通行可能タイルの区別は現行コードにある
- `未実装`: 子ども向けに森づくりを専用チュートリアルとして支える仕組みはない

```gherkin
Feature: CJG03 森を作る
  子どもが森を自分で設計するには
  木を置いた結果が見た目にも移動にも
  一貫して伝わらなければならない
```

```gherkin
Rule: 木が集まると通れない森になる
  見た目だけ森で通れてしまうと、地形の意味が伝わらない

  Scenario: 木タイルを密集させると通行不可エリアができる
    Given マップ上に草タイルのエリアがある
    When 木タイルを複数マスにまとめて配置する
    And Run する
    Then キャラは木タイルの上を歩けない
    And 木タイルの隙間にある草タイルは歩ける
```

```gherkin
Rule: 通り道を残さないと閉じ込められることが自分で分かる
  子どもが「出口を作らないと出られない」と遊びながら理解できなければならない

  Scenario: 木で完全に囲むと閉じ込められる
    Given 木タイルでエリアを完全に囲む
    When Run してキャラがそのエリアに入る
    Then キャラはエリアの外に出られない
```

## CJG04: 水辺を作る

この `CJG04` では、「水と岸を使って、ひと目で分かる湖や水辺を作れる」を約束する。

実装状況:
- `部分実装`: 水タイル、岸の見た目、通行不可判定は現行コードにある
- `未実装`: Code Maker / Resource Editor で見える水辺と実ゲームの一致はまだ保証できていない

```gherkin
Feature: CJG04 水辺を作る
  子どもが湖や水辺を自分で作るには
  水の見た目と通れなさが一致し
  岸と組み合わせてまとまって見えなければならない
```

```gherkin
Rule: 水タイルは水として見えて水として通れない
  水なのに歩けると、地形のルールが子どもに伝わらない

  Scenario: 水タイルは通行不可として機能する
    Given マップ上に草タイルのエリアがある
    When 水タイルを配置する
    And Run する
    Then キャラは水タイルの上を歩けない
```

```gherkin
Rule: 水と岸を組み合わせて湖に見える
  単なる青い四角の集まりではなく、水辺だと分かるまとまりが必要である

  Scenario: 水タイルと岸タイルで湖を作れる
    Given タイルパレットに水タイルと岸タイルがある
    When 水タイルで内側を、岸タイルで縁を配置する
    And Run する
    Then 湖として視覚的にまとまって見える
    And 岸タイルも通行不可である
```

## CJG05: 装飾で世界を彩る

この `CJG05` では、「花や岩を置くと、通りやすさはそのままに、世界だけがにぎやかになる」を約束する。

実装状況:
- `実装済み`: 装飾タイルと遭遇率は現行コードに定義されている
- `部分実装`: ゾーンごとの装飾方針はコードにあるが、子ども向け専用導線はない

```gherkin
Feature: CJG05 装飾で世界を彩る
  子どもが世界を自分らしく飾るには
  装飾で見た目の個性を足せて
  基本の遊びやすさを壊してはいけない
```

```gherkin
Rule: 装飾は見た目を変えても歩きやすさを壊さない
  花や岩を置いただけで急に通れなくなると、気軽に飾れない

  Scenario: 装飾タイルは通行可能である
    Given マップ上に装飾タイルを配置する
    When Run する
    Then キャラは装飾タイルの上を歩ける
    And エンカウント率は草タイルと同じである
```

```gherkin
Rule: ゾーンごとに飾り分けて場所の個性を出せる
  どこも同じ見た目だと、子どもが作った世界の違いが弱くなる

  Scenario: 装飾タイルをゾーンごとに使い分けられる
    Given ゾーン0に花タイルを配置する
    And ゾーン1にキノコタイルを配置する
    And ゾーン3にサボテンタイルを配置する
    When Run する
    Then 各ゾーンが異なる見た目になる
```

## CJG06: 迷路を作る

この `CJG06` では、「壁と道を組み合わせて、自分で解ける迷路を作れる」を約束する。

実装状況:
- `部分実装`: 迷路を作れるだけの通行判定とダンジョン tilemap はある
- `未実装`: 迷路の完成判定や出口チェックを支援する専用機能はない

```gherkin
Feature: CJG06 迷路を作る
  子どもが迷路づくりを楽しむには
  壁と道のルールが分かりやすく
  自分で遊んで確かめられなければならない
```

```gherkin
Rule: 壁と道の役割がはっきり分かれる
  壁なのに抜けられる、道なのに進めない、を作ってはいけない

  Scenario: 壁タイルと道タイルで迷路が作れる
    Given 壁タイルと通行可能な道タイルがパレットにある
    When 壁と道を組み合わせて迷路パターンを作る
    And Run する
    Then キャラは道タイルの通路のみ移動できる
    And 壁タイルで行き止まりを作れる
```

```gherkin
Rule: 作った迷路は自分で解いて確かめられる
  作るだけでなく、自分で入口から出口まで試せなければならない

  Scenario: 迷路をテストプレイして解けるか確認できる
    Given 迷路パターンを Tilemap で作成した
    When Run してキャラを操作する
    Then 入口から出口まで到達できるか自分で確認できる
    And 行き止まりに入ったら引き返せる
```

## CJG07: ランドマークを配置する

この `CJG07` では、「世界樹や塔のような特別な場所を置くと、その場所が特別だとすぐ分かる」を約束する。

実装状況:
- `部分実装`: 現在の実装には世界樹と通信塔の固定ランドマーク、強調表示、近接イベントがある
- `未実装`: Editor 上で 2x2 ランドマークを置くと自動配置される一般化機能はまだない

```gherkin
Feature: CJG07 ランドマークを配置する
  子どもが世界に特別な場所を作るには
  大きな建物や木が正しい形で置けて
  行ける場所と触る場所が分かりやすくなければならない
```

```gherkin
Rule: 大きいランドマークは正しい形で配置される
  2x2 の建物が欠けたりずれたりすると、特別な場所に見えない

  Scenario: マルチタイルランドマークを配置できる
    Given タイルパレットに世界樹や通信塔のランドマークタイルがある
    When ランドマークタイルをマップに配置する
    Then 2x2のタイルが正しい位置関係で配置される
```

```gherkin
Rule: ランドマークは乗る場所ではなく触る場所として伝わる
  上に乗れてしまうと、特別な場所としての意味が薄れる

  Scenario: ランドマークは通行不可で隣接から調べられる
    Given マップにランドマークが配置されている
    When Run してキャラをランドマークに向かって歩かせる
    Then ランドマークの上には乗れない
    And ランドマークに隣接した状態でインタラクションするとイベントが発生する
```

```gherkin
Rule: ランドマークは既存の大事な場所と重ならない
  城や町と重なると、子どもが世界地図を理解しにくくなる

  Scenario: ランドマーク配置が既存の拠点と重ならない
    Given CASTLE_POS, TOWN_*, CAVE_GLITCH の座標がある
    When ランドマークを配置する
    Then 既存拠点の座標とランドマークのタイルが重ならない
```

## 共通条件

この共通条件では、「Tilemap エディタで見えることが真実であり、内部変換がそれを裏切らない」を約束する。

実装状況:
- `部分実装`: 逆変換と tile bank の整合は現行コードにある
- `未実装`: `Editor 表示 = 実ゲーム表示` を end-to-end で保証できていない

```gherkin
Feature: 共通条件 Map
  子どもが安心してマップを編集するには
  編集画面とゲーム画面が一致し
  内部変換がその体験を壊してはいけない
```

```gherkin
Rule: Tilemap エディタで見える形がゲーム世界の真実である
  親の説明がないと分からない隠れた変換は、この製品の価値に反する

  Scenario: Tilemap エディタの結果がそのままゲームに反映される
    Given Tilemap エディタでタイルを変更した
    When Run する
    Then ゲーム内の地形と通行可否は Tilemap エディタで見えた内容と一致する
    And 子どもに見えない runtime 専用の地形補正に依存しない
```

```gherkin
Rule: 内部変換は保存や実行のために使っても、見える結果を変えてはいけない
  変換が必要でも、子どもが見たものと別の世界を作ってはならない

  Scenario: derive_world_from_tilemap が GUI 編集結果を壊さない
    Given Tilemap エディタでタイルを変更した
    When derive_world_from_tilemap が内部データへ変換する
    Then 変換後のデータでゲームが正常に動作する
    And 変換前に子どもが見た地形の意味と見た目が保たれる
    And 既存のセーブデータとの互換性が維持される
```

```gherkin
Rule: ゲーム内で使うすべてのタイルはエディタでも扱える
  見えるけれど編集できない、または編集できるのにゲームで壊れる、を作ってはいけない

  Scenario: すべてのタイルIDがタイルバンクに登録されている
    Given ゲーム内で定義されたタイルID一覧がある
    Then すべてのタイルIDがタイルバンクに登録されている
    And 各タイルの16x16ピクセルデータが Pyxel リソースに描画されている
```
