# カスタマージャーニーgherkin: Battle（戦闘・敵・呪文・バランス）

- 対象カスタマージャーニー: CJ08, CJ10, CJ13, CJ29
- 根拠: [`customer-journeys.md`](./customer-journeys.md)
- 中心テーマ: 敵データ, 呪文データ, 戦闘ロジック, バランス調整
- 読み方: `Feature` は何を約束するか, `Rule` は何を壊してはいけないか, `Scenario` は画面や動作でどう確かめるか
- 実装状況の読み方: `実装済み` は現行コードとテストで確認できる, `部分実装` はコアはあるが文中の約束を全部は満たしていない, `未実装` は目標, `時間目標` は現状の目標値であり自動保証ではない

## CJG08: 敵が強すぎる

この `CJG08` では、「強すぎる敵をすぐ調整できて、直したせいで別の仕組みが壊れない」を約束する。

実装状況:
- `部分実装`: 敵データは `assets/enemies.yaml` と `src/game_data.py` から読み込まれ、戦闘やレベルアップの基本ロジックにはテストがある
- `時間目標`: `5分以内` は UX 目標であり、現状コードでは自動計測していない

```gherkin
Feature: CJG08 敵が強すぎる
  子どもがちょうどよい手ごたえで遊ぶには
  敵の強さをすぐ調整できて
  その変更で戦闘システム全体が壊れてはいけない
```

```gherkin
Rule: 強さの変更はすぐテストプレイへ反映される
  変えた結果がすぐ見えないと、親子の調整ループが止まる

  Scenario: 敵のHPを変更するとテストプレイで即座に反映される
    Given ENEMIES データに「スライム」が HP=50 で定義されている
    When 親がAIに「スライムのHPを30にして」と頼む
    And AIがコードを修正する
    And Run する
    Then スライムとの戦闘でHPが30として表示される
    And 修正から確認まで5分以内で完了する
```

```gherkin
Rule: 数値を変えても戦闘の基本は壊れない
  HPだけ直して報酬やレベルアップが壊れると、調整した意味がなくなる

  Scenario: 敵のパラメータ変更が他のゲームシステムを壊さない
    Given 敵のHP/ATK/DEF/EXP/GOLDを変更した
    When Run してその敵と戦闘する
    Then ダメージ計算が正しく動作する
    And 勝利時に正しいEXPとGOLDが得られる
    And レベルアップ判定が正常に動作する
```

## CJG10: 新しい敵を追加したい

この `CJG10` では、「新しい敵を足したら、ちゃんと出現して、ちゃんと見えて、ちゃんと戦える」を約束する。

実装状況:
- `部分実装`: 敵データの追加、ゾーン別出現、イベント敵の除外は現行データ構造にある
- `部分実装`: スプライトは `sprite` 参照で扱うが、新規絵の作成自体は Code Maker 側の作業になる

```gherkin
Feature: CJG10 新しい敵を追加したい
  子どもが新しい敵との出会いを楽しむには
  敵データを追加できて
  指定した場所で出現し
  見た目も戦闘画面に出なければならない
```

```gherkin
Rule: 新しい敵は必要な情報をそろえて追加できる
  必須項目が欠けた敵は、出現してもまともに動かない

  Scenario: 新しい敵をENEMIESデータに追加できる
    Given ENEMIESデータの構造（name, sprite, hp, atk, def, agi, exp, gold, zone, category, desc）がある
    When 新しい敵「ドラゴン」を追加する
    Then ENEMIESデータにドラゴンが含まれる
    And 必須フィールドがすべて埋まっている
```

```gherkin
Rule: 敵は出したい場所にだけ出る
  どこでも出ると、場所ごとの難しさや驚きが崩れる

  Scenario: 追加した敵が指定ゾーンでエンカウントする
    Given 「ドラゴン」をゾーン3に出現するよう設定した
    When Run してゾーン3を歩く
    Then ドラゴンとのエンカウントが発生する
    And ゾーン0-2ではドラゴンが出現しない
```

```gherkin
Rule: 新しい敵は戦闘画面で見える
  データだけ増えても見た目が出なければ、子どもには追加した感じが伝わらない

  Scenario: 追加した敵のスプライトがゲーム内で表示される
    Given 「ドラゴン」のスプライトをスプライトエディタで描いた
    When ドラゴンとの戦闘に入る
    Then 戦闘画面にドラゴンのスプライトが表示される
```

## CJG13: 新しい呪文がほしい

この `CJG13` では、「新しい呪文を足したら、戦闘中にちゃんと使えて、使えない時も理由が分かる」を約束する。

実装状況:
- `部分実装`: 現在の呪文スキーマは `name, mp, type, power, desc, learn_lv` で、`type` は `heal/attack` の2種類
- `実装済み`: MP不足時のメッセージ、レベル到達時の習得、戦闘中の使用は現行コードにある
- `未実装`: 任意ターゲット種別や汎用的な呪文ビルダーはまだない

```gherkin
Feature: CJG13 新しい呪文がほしい
  子どもが新しい作戦を楽しむには
  呪文を追加できて
  戦闘中に使えて
  使えないときも理由が伝わらなければならない
```

```gherkin
Rule: 新しい呪文は必要な情報をそろえて追加できる
  名前だけあっても、消費MPや威力がなければ戦闘で使えない

  Scenario: 新しい呪文をSPELLSデータに追加できる
    Given SPELLSデータの構造（name, mp, type, power, desc, learn_lv）がある
    When 新しい呪文「ファイアボール」を追加する
    Then SPELLSデータにファイアボールが含まれる
    And 必須フィールドがすべて埋まっている
```

```gherkin
Rule: 追加した呪文は戦闘で正しく動く
  メニューには見えても使えない、ダメージだけ出てMPが減らない、のようなズレを作ってはいけない

  Scenario: 追加した呪文が戦闘中に使用できる
    Given 「ファイアボール」がSPELLSに追加されている
    And プレイヤーが十分なMPを持っている
    When 戦闘中に呪文メニューを開く
    Then 「ファイアボール」が選択肢に表示される
    When ファイアボールを選択する
    Then ダメージが計算され敵に適用される
    And MPが消費される
```

```gherkin
Rule: 使えない時は理由が分かる
  ただ失敗するだけだと、子どもは何が足りないか分からない

  Scenario: MP不足で呪文が使えない場合のフィードバック
    Given プレイヤーのMPがファイアボールのmp未満
    When 戦闘中にファイアボールを選択する
    Then 「MPがたりない」旨のメッセージが表示される
    And MPは消費されない
```

## CJG29: 全体のバランスを調整する

この `CJG29` では、「後半だけ強くする、成長を少しゆっくりにする、のような全体調整をしてもゲームが最後まで遊べる」を約束する。

実装状況:
- `部分実装`: 敵・呪文・成長データは YAML から変えられる
- `未実装`: ゾーン単位の一括倍率調整を製品機能として案内する UI や専用コマンドはない
- `時間目標`: 外部フィードバックからその場で何回も回す速度目標は、現状コードでは保証していない

```gherkin
Feature: CJG29 全体のバランスを調整する
  子どもが最後までちょうどよい難しさで遊ぶには
  敵や成長のカーブをまとめて調整できて
  そのあともゲームクリアまで破綻なく遊べなければならない
```

```gherkin
Rule: 指定した範囲だけをまとめて調整できる
  後半だけ難しくしたいのに前半まで変わると、狙った調整にならない

  Scenario: 敵パラメータを一括で倍率調整できる
    Given 複数の敵がENEMIESデータに定義されている
    When 親がAIに「ゾーン3以降の敵のHP/ATKを1.5倍にして」と頼む
    And AIがデータを修正する
    Then ゾーン3以降の敵のHP/ATKが元の1.5倍になっている
    And ゾーン0-2の敵は変更されていない
```

```gherkin
Rule: 調整後も通しプレイが成立する
  難しくしすぎたり、育ちにくくしすぎたりして詰んではいけない

  Scenario: バランス調整後にゲームが正常に動作する
    Given 敵パラメータとEXPカーブを一括調整した
    When Run して最初から通しプレイする
    Then 各ゾーンで適切な難易度の戦闘が発生する
    And レベルアップのペースが極端に速すぎたり遅すぎたりしない
    And ゲームクリアまで到達可能である
```

```gherkin
Rule: 外からの感想でバランスを回せる
  親子だけでなく友達の感想からも、難しさ調整を回せなければならない

  Scenario: 友達のフィードバックからバランス調整が回る
    Given 友達がスマホでゲームをプレイして「後半簡単すぎる」と言った
    When 親がAIに「ゾーン3以降の敵を強くして」と頼む
    And 修正後にWebビルドを再実行する
    Then 友達がスマホで修正版を確認できる
    And 「ちょうどいい」か「まだ簡単」のフィードバックがその場で返る
```

## 共通条件

この共通条件では、「戦闘データをいじっても、最低限の整合性が崩れない」を約束する。

実装状況:
- `実装済み`: `test/test_game_data.py` と `test/test_spell_logic.py` が現行スキーマと基本挙動を確認している

```gherkin
Feature: 共通条件 Battle
  戦闘まわりの調整を安心して続けるには
  敵と呪文のデータが
  最低限の条件を満たしていなければならない
```

```gherkin
Rule: 敵データは戦闘に使える形を保つ
  体力や報酬が0以下の敵は、RPGとしての手ごたえを壊す

  Scenario: ENEMIESデータの整合性
    Given ENEMIESデータに定義されたすべての敵
    Then 各敵のHP > 0, ATK > 0, DEF >= 0, EXP > 0, GOLD >= 0 である
    And zone の値が有効なゾーン番号である
```

```gherkin
Rule: 呪文データは戦闘に使える形を保つ
  消費MPや威力や対象が壊れていると、戦闘メニューが意味を失う

  Scenario: SPELLSデータの整合性
    Given SPELLSデータに定義されたすべての呪文
    Then 各呪文のmp > 0, power > 0 である
    And type が heal または attack である
    And learn_lv が昇順に並んでいる
```
